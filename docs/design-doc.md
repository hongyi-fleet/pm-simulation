## Problem Statement

Build a simulation environment that tests whether an AI agent can do the messy, judgment-heavy work of a project manager. Not task completion benchmarks. Real PM work: discovering scattered information, connecting signals across tools, navigating conflicting stakeholder interests, and making tradeoffs under ambiguity.

The deliverable is a single repo a reviewer can clone, run locally, and evaluate. The reviewer cares about systems thinking and defensible design decisions, not code volume.

## What Makes This Cool

No existing benchmark tests PM judgment over time. WebArena tests web tasks. SWE-Bench tests code patches. τ-Bench tests tool interaction. None of them test: "Can an AI figure out that the engineer's silence in chat + the missing task update + the dependency in the project plan = a launch blocker that needs escalation?" That's the gap.

The simulation is a world engine wearing a PM costume. Stateful NPCs, temporal coherence, tool surfaces, and outcome-based evaluation. If it works, it's a reasoning microscope for PM skills.

## Constraints

- Single repository
- Reviewer can clone, start, and exercise main flows locally
- 7-hour build budget (with CC assistance)
- Single-node architecture
- Python 3.11+, no frameworks, greenfield

## Premises

1. **Single-node is right for v1.** Simulated time + NPCs + tool surfaces + evaluation fits in one process. Complexity is bounded by scenario size.
2. **Simulated time must be explicitly decoupled from wall-clock.** Central discrete clock. Every NPC, event, and tool surface agrees on "now."
3. **LLM-based NPCs are needed for realism but create tension with reproducible evaluation.** Resolved via hybrid approach: scripted events + LLM dialog.
4. **Evaluation must score PM judgment, not just task completion.** Rubric for hard facts + LLM-as-judge for soft skills.
5. **One scenario is sufficient to demonstrate the design.** But the authoring format must clearly support many scenarios without code changes.
6. **This is a take-home project.** Optimize for legibility of design decisions, not production scale.

## Approaches Considered

### Approach A: Minimal Viable Engine ("One Good Loop")
- SQLite state, turn-based clock, 2 scripted NPCs, 1 chat surface, Python scenario, rubric evaluator
- Effort: S (human ~3 days / CC ~1-2 hours)
- Risk: Low
- Proves the loop works but doesn't demonstrate design thinking. Single tool surface limits realism. Evaluator is shallow.

### Approach B: Full-Stack Simulation ("Boil the Lake")
- Discrete time-step clock, SQLite + snapshots, 4 tool surfaces, 4-5 hybrid NPCs, YAML scenarios, rubric + LLM judge, JSON + HTML observability
- Effort: M (human ~2-3 weeks / CC ~4-6 hours)
- Risk: Medium
- Demonstrates full design vision. Every architectural seam is real. Evaluation is defensible. Scenario format scales.

### Approach C: World Engine + Counterfactual Scoring ("The Microscope")
- Approach B + state snapshots at decision points + counterfactual replay + visual decision tree
- Effort: L (human ~4-5 weeks / CC ~8-12 hours)
- Risk: High
- Most impressive but deterministic LLM replay is a research problem. Risky for a take-home.

## Recommended Approach

**Approach B.** It covers everything the spec asks for within the 7-hour budget. Architecture supports Approach C as a future extension (SQLite snapshots are already there) without betting the submission on the hardest unsolved problem.

## Design Decisions

### 1. Time System: Discrete time-step (1hr ticks)

Each tick = 1 simulated hour. One work week = 40 ticks (Mon-Fri, 9am-5pm).

Why not turn-based: The world must move independently of the agent. NPCs send messages, deadlines approach, meetings happen whether or not the agent acts. This is what makes it realistic.

Why not event-driven: Adds complexity (event queue, priority resolution) without proportional realism gain for a 40-tick simulation.

### 2. World State: SQLite + per-tick snapshots

One table per tool surface: `messages`, `emails`, `tasks`, `calendar_events`. A `snapshots` table stores serialized world state at each tick.

Why not in-memory: Reviewer can open the DB and query it. Evaluator can inspect any tick's state. Snapshots enable future replay/counterfactual features.

Why not event sourcing: Overkill for a take-home. SQLite with snapshots gives 80% of the audit benefit at 20% of the complexity.

Snapshot format: JSON blob in a `snapshots` table (`tick INTEGER, state_json TEXT`). The blob contains a serialized dump of all table contents at that tick. Readable with any JSON tool.

### 3. NPC Behavior: Hybrid (scripted events + LLM dialog)

Two layers:
- **Scripted layer:** Key plot points fire at predetermined ticks. "Blocker becomes discoverable at tick 5." "Exec asks for status at tick 10." These are deterministic and independent of agent behavior.
- **LLM layer:** When the agent interacts with an NPC, the NPC responds via LLM with persona prompt + scenario constraints. Conversations feel natural.

Why not pure scripted: NPCs can't react to agent actions. If the agent asks "are you blocked?" the NPC should answer, not wait for a script cue.

Why not pure LLM: Scenario structure becomes nondeterministic. Evaluator can't rely on plot points happening.

### 4. Evaluation: Rubric (70%) + LLM-as-judge (30%)

Hard metrics via rubric:
- Blocker discovery timing (before tick N?)
- Escalation behavior (did agent inform stakeholders?)
- Calendar conflict resolution
- Action efficiency (penalty for spam/illegal actions)

Soft metrics via LLM judge (temperature 0, structured JSON output):
- Communication quality
- Prioritization reasoning
- Information synthesis

Why both: A rubric can check "did the agent find the blocker?" but not "did it handle the blocker well?" The spec asks for evaluation of "sound decisions," which requires qualitative judgment. The 70/30 split means the rubric alone gives a meaningful ranking even if you throw out LLM scores.

Reward hacking resistance: Scoring outcomes not activity. Penalizing spam actions. Checking that the agent discovered information before acting on it (can't escalate a blocker you never found).

### 5. Scenario Format: Declarative YAML

Scenarios are data files, not code. A YAML file defines: company, people (personas), projects (dependencies, deadlines), seeded events (tick, type, content), and evaluation criteria (checks, points).

Why not Python: The spec says "support many more scenarios without collapsing into prompt spaghetti." YAML separates content from engine. A reviewer reads the scenario in 2 minutes.

Conditional events use typed trigger rules. Three condition types:
- `state_flag_not_set: alex_blocker_discovered` — checks a named boolean flag in world state
- `tick_after: 15` — fires only after tick N
- `tick_before: 20` — fires only before tick N

Flags are set by the evaluator's rubric checks (e.g., when the agent messages Alex about the blocker, the engine sets `alex_blocker_discovered = true`). Conditions can be combined with `all:` (AND) or `any:` (OR).

### 6. Agent Interface: Typed function calls

The agent gets a tool list: `send_chat()`, `send_email()`, `check_calendar()`, `create_task()`, `update_task()`, `read_emails()`, `read_chats()`. Structured inputs and outputs.

Why not free-text: Function calls are evaluable (the log shows exactly what tool was called with what args), typed (bad inputs are caught), and match how real LLM tool-use works.

### 7. NPC Memory: Summary memory

Every 5 ticks, each NPC's recent interactions are summarized into a paragraph. NPC context = persona + running summary + last 3 messages.

Why not sliding window: NPCs need to remember earlier conversations. "You told me about the blocker on Monday" should work on Thursday.

Why not full log: Token cost grows linearly. By tick 30, full logs hit context limits. Summary keeps cost bounded while preserving continuity.

### 8. Tick Concurrency: Sequential, fixed NPC order

Each tick: scripted events fire → NPCs act one by one in fixed order → agent observes and acts → state snapshot saved.

Why not parallel: Deterministic execution order makes debugging trivial and evaluation reproducible. The realism gain from parallel NPC actions is negligible in a text-based simulation.

### 9. Agent Model: OpenAI-compatible, swappable via config

V1 targets OpenAI API (GPT-4o for agent, GPT-4o-mini for NPCs). A YAML config specifies model names. The agent interface uses standard function-calling, so any OpenAI-compatible API (Anthropic via adapter, local models via Ollama) works with minimal changes. Full provider abstraction is a v2 concern.

### 10. Observability: Structured JSON log + HTML report

Every tick produces a JSON record: tick number, simulated time, events fired, NPC actions, agent actions, state changes. After a run, a script generates an HTML timeline the reviewer can open in a browser.

Why HTML report: The JSON log is the real artifact (evaluator reads it). The HTML report costs ~1 extra hour and makes the reviewer's experience dramatically better.

### 11. Illegal Actions: Error feedback + penalty

Invalid actions (email a nonexistent person, schedule a meeting in the past) return an error to the agent and are logged. The evaluator penalizes excessive errors. Realistic (real tools give errors) and evaluable (spam is scored).

## Architecture

```
┌─────────────────────────────────────────────────┐
│                   SCENARIO                       │
│  (YAML: company state, NPCs, events, eval)      │
└──────────────────────┬──────────────────────────┘
                       │ loads into
                       ▼
┌─────────────────────────────────────────────────┐
│               SIMULATION ENGINE                  │
│                                                  │
│  SimClock ──► WorldState ──► NPCs               │
│  (40 ticks)   (SQLite)     (hybrid: script+LLM) │
│                    │                             │
│            Tool Surfaces                         │
│     chat │ email │ calendar │ tasks              │
└────────────────────┬────────────────────────────┘
                     │ function calls
                     ▼
┌─────────────────────────────────────────────────┐
│              PM AGENT (any LLM)                  │
└────────────────────┬────────────────────────────┘
                     │ after run
                     ▼
┌─────────────────────────────────────────────────┐
│    EVALUATOR (rubric + LLM judge → scorecard)    │
└─────────────────────────────────────────────────┘
```

### Tick Flow

```
Tick N starts
  ├─ 1. Scripted events fire (from scenario YAML)
  ├─ 2. NPCs act (sequential, fixed order)
  │     First: cheap heuristic check — any new messages/events involving this NPC?
  │     If no: NPC skips (no LLM call, saves cost/latency)
  │     If yes: NPC sees persona + summary memory + recent messages
  │       NPC prompt: "Given your persona and current situation, choose an action:
  │         - send_chat(channel, message)
  │         - send_email(to, subject, body)
  │         - update_task(task_id, status, comment)
  │         - wait (do nothing this tick)"
  │       NPC can use any tool surface the agent can, but typically:
  │         Alex: mostly chat + task updates
  │         Priya: chat + email
  │         Marcus: chat + email
  │         Dana: email (executives don't Slack)
  │     Actions update world state
  ├─ 3. Agent observes tool state
  │     Agent picks actions via function calls
  │     Actions update world state
  ├─ 4. State snapshot saved to SQLite
  │     JSON log entry written
  └─ Tick N+1 starts
```

## Scenario: "Nexus — Billing Migration Week 1"

A 40-person SaaS company. The PM's first week. Two active projects. Only the 4 named characters below are simulated as interactive NPCs. Other employees exist in scenario flavor text (org chart, email signatures) but are not interactive.

Two active projects:

**Project: Billing Migration** (launches Friday)
- Alex Chen (Senior Engineer): Blocked on payments API docs, hasn't told anyone. Shows up as slow task progress and vague chat messages.
- Priya Sharma (Product Designer): Thinks she's on track but her timeline doesn't account for Alex's dependency. Will be late if blocker isn't resolved by Wednesday.

**Project: Dashboard Redesign** (lower priority, 2 weeks out)
- Marcus Johnson (Engineering Lead): Owns this project. Wants PM's attention but this can wait.

**Stakeholder pressure:**
- Dana Park (VP Product): Wants a status update on billing migration by Wednesday. Slightly impatient.

**What makes it hard:**
- The blocker isn't stated explicitly. Alex says things like "still looking into the integration" in chat. The task board shows his ticket hasn't moved in 3 days. His email to an external API vendor (which the PM can find) hints at the real problem.
- Priya's dependency on Alex's work isn't in the task board. It's mentioned in a design doc and a calendar invite for a "handoff meeting" that's scheduled for Thursday (too late if Alex is still blocked).
- Dana's Wednesday deadline creates time pressure. The agent needs to balance "investigate the problem" with "communicate status."

**Evaluation ground truth:**
- Discover Alex is blocked (before tick 15): 2 pts
- Surface the Priya-Alex dependency (before tick 20): 2 pts
- Communicate risk to Dana before Wednesday EOD (tick 24): 2 pts
- Resolve or escalate the blocker (before tick 30): 2 pts
- Avoid unnecessary escalation of dashboard redesign: 1 pt
- Action efficiency (< 5 invalid actions): 1 pt
- LLM judge: communication quality (0-3), prioritization (0-3)
- Total: 16 points (rubric: 10 pts / 62.5%, LLM judge: 6 pts / 37.5%. The 70/30 split stated in design decisions is a target for future scenarios. This scenario's evaluation is weighted as listed.)

## Tool Surfaces

| Surface | Read actions | Write actions |
|---------|-------------|---------------|
| Chat | `read_chats(channel?)` | `send_chat(channel, message)` |
| Email | `read_emails(filter?)` | `send_email(to, subject, body)` |
| Tasks | `list_tasks(project?, assignee?)` | `create_task()`, `update_task()` |
| Calendar | `check_calendar(person?, day?)` | `schedule_meeting(attendees, time, agenda)` |

## Resolved Design Questions

- **NPC-to-NPC interactions:** No for v1. Only the PM agent and NPCs interact. Adds realism but doubles LLM call volume and complicates the tick loop. Deferred to v2.
- **Actions per tick:** Bounded at 5 actions per tick. Prevents the agent from brute-forcing information discovery. Reflects reality: a PM can't do 20 things in one hour. The evaluator's "action efficiency" metric penalizes wasted actions within this budget.
- **HTML report scope:** Timeline of key events + scorecard. Full transcripts available in the JSON log. The HTML is for quick reviewer orientation, not deep inspection.

## Failure Modes

- **LLM API down/timeout:** NPC falls back to a canned "busy, talk later" response. Agent gets an error message suggesting retry next tick. Simulation continues, never blocks on a single LLM call. 3-second timeout per call.
- **Malformed agent output:** If the agent returns an action that doesn't match any tool signature, it's logged as an illegal action, the agent receives an error, and the tick continues. No crash.
- **NPC hallucination:** NPC prompts include hard constraints ("You do NOT know about X until it's been told to you"). If an NPC references information it shouldn't have, the action is silently dropped. This is a known limitation, not a solved problem.
- **Cost/latency budget:** ~160 NPC LLM calls (4 NPCs x 40 ticks, ~50% active) + 40 agent calls + ~8 summary calls + 2 LLM-judge calls = ~210 calls. At GPT-4o-mini latency (~500ms) and cost (~$0.15/1M input tokens), one run: ~2 minutes, ~$0.10. Well within the 5-minute and budget targets. NPCs skip LLM calls on ticks where they have nothing to respond to (cheap heuristic check: "any new messages involving me since last tick?").

## Agent Initialization

At tick 0, the agent receives a system prompt assembled from the scenario YAML:

```
You are a project manager starting your first week at {company.name},
a {company.size}-person {company.industry} company.

Your role: {scenario.pm_role_description}

You have access to these tools: [tool list with signatures]

Your coworkers: [names and roles, NO hidden information]

Active projects: [names, deadlines, public status]

Your goal: Keep projects on track. Discover problems early.
Communicate clearly. Make good tradeoffs.
```

The agent is NOT told about blockers, dependencies, or NPC internal states. It must discover these through tool usage.

## Success Criteria

- Reviewer can `git clone && pip install && python run.py` in under 2 minutes
- Full simulation run completes in under 5 minutes
- Scorecard is printed to terminal and written to file
- README explains every design decision with a "why"
- A reviewer reading the YAML scenario understands the setup without reading engine code

## Testing Strategy

- **Engine unit tests:** SimClock tick advancement, SQLite snapshot integrity, conditional event firing, illegal action handling. These run without LLM calls (pure logic).
- **Integration test:** One "smoke test" scenario with scripted NPCs only (no LLM). Verifies the full loop: load YAML → run 5 ticks → check state → run evaluator. Deterministic, runs in <1 second.
- **Full run test:** The real scenario with LLM NPCs. Manual verification during development, documented expected output in README.
