# Tradeoffs & TODOs

Decisions made, limitations acknowledged, and future work prioritized.

---

## Tradeoffs

### T1: Neither agent nor NPCs need memory summarization at current scale

**Decision:** Agent uses a sliding window (first 2 + last 16 messages). NPCs have summary memory every 5 interactions but it's only 150 tokens. Neither actually needs summarization.

**Data:**
- Agent: 1 week generates ~8,000 tokens of tool data. 6.4% of 128K context. Agent reads from SQLite, not conversation history. Sliding window drops old reasoning, not source data.
- NPC: Context uses 335-485 tokens out of 2,000 budget (17-24%). State progression + last 3 messages handles consistency.

**Threshold:** Summarization needed at ~13 weeks, ~53 NPCs, or ~10 projects.

**If challenged:** "8,000 tokens out of 128K. The database is the agent's memory. NPC uses 17% of budget. State progression handles consistency. We have the NPC summarization implemented as future-proofing, but the data shows it's not needed at this scale."

### T2: NPC state progression is day-level, not hour-level

**Decision:** Alex's hidden state changes per day (Mon/Tue/Wed/Thu/Fri), not per hour.

**Why:** Day boundary is a natural social rhythm. Hour-level adds complexity without proportional gain for a 5-day simulation.

**Downside:** Within Monday, Alex is static. 9am and 4pm get the same response.

**If challenged:** "Day-level for v1. YAML supports finer granularity — add `start_hour`/`end_hour`. Engine needs minor change to `get_current_hidden_state`. Not blocked architecturally."

### T3: Agent uses cooldown to prevent cascade

**Decision:** Agent has three trigger modes:
1. **Scheduled patrol** every 30 minutes (proactive investigation)
2. **NPC reply triggers** agent turn (reactive), but with 10-minute cooldown
3. **Structural events** directed at agent trigger immediately (Dana's email)

After any agent turn, a 10-minute cooldown starts. NPC replies during cooldown are accumulated. When cooldown ends, one agent turn processes all accumulated messages.

**Why cooldown:** Without it, we observed cascade explosion. Agent sends 4 messages → 4 NPC replies → each triggers agent turn → agent sends 4 more → 796 events in one run instead of ~100. Root cause: agent + NPC feedback loop with no damping.

**Why not pure batch (no NPC-triggered turns at all):** Agent would wait up to 30 minutes to see any reply. A PM checks Slack when they get a notification, not on a timer. Cooldown preserves responsiveness while preventing cascade.

**Why not immediate (no cooldown):** Tested this. Cascade exploded. 414 agent turns in one week, 357 chat messages. Agent spammed everyone because every NPC reply gave it a new turn to act.

**Architecture is event-driven + periodic polling.** The 80 scheduled turns look like a clock, but everything else is event-driven: NPC responses at irregular times, structural events at specific timestamps, cooldown-batched reactive turns.

**If challenged:** "We tested three modes: pure batch (30-min delay, no cascade), immediate response (zero delay, cascade explosion), and cooldown (10-min damping, no cascade). Cooldown is the sweet spot. It's configurable in the YAML (`agent_cooldown_minutes`). The cascade problem is real — we measured it: 796 events vs ~150 expected."

### T4: No NPC-to-NPC interactions

**Decision:** NPCs only interact with the PM agent, not with each other.

**Why:** Doubles LLM call volume. Creates complex interaction chains. Not needed for the core evaluation.

**Downside:** Missing an information discovery channel. In reality Priya might ask Alex about the handoff in #general. PM would see it.

**If challenged:** "Architecture supports it — NPCs can send_chat to any channel. Just need NPC-to-NPC activation logic in Game Master. This is v2."

### T5: Two-layer signal detection has LLM nondeterminism

**Decision:** Layer 1 is deterministic (SQL state check). Layer 2 is LLM-as-judge. Same pattern as TheAgentCompany (18% of their 175 tasks use LLM eval).

**Why:** "Did the agent learn Alex is blocked?" requires understanding conversation meaning, not just checking if a message was sent.

**Downside:** LLM judge might give different answers across runs.

**If challenged:** "Layer 1 gates Layer 2 — LLM fires ~5 times per run, not 80. Temperature 0. Can run 3x and take majority vote. Variance is measured via `--runs N`, not hidden."

### T6: Meetings are LLM-generated, not scripted

**Decision:** Standup transcripts are generated by calling each NPC's LLM with their current state progression.

**Why:** Scripted meetings contradict "no hardcoded NPCs." Monday Alex says "fine." Thursday Alex reveals the problem. Reflects evolving state.

**Downside:** Meeting content varies per run. Monday standup might accidentally reveal too much.

**If challenged:** "State progression controls what they're WILLING to say. LLM controls HOW. Monday prompt says 'thinks he'll figure it out.' If the LLM generates a confession, that's prompt engineering, not architecture."
