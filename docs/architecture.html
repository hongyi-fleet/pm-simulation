2. The Event Loop

Time doesn't tick. Events drive everything. Discrete Event Simulation, the industry standard for workplace modeling.

1
Pop next event from queue
Priority queue sorted by simulated time. Same-time events: structural > NPC > agent.
2
Advance clock
Passive SimClock jumps to event time. If nothing happens for 2 hours, skip 2 hours.
3
Game Master resolves event
Structural events write to tool surfaces. NPC responses trigger LLM calls. Concordia-inspired pattern.
4
NPCs react (with realistic delays)
Alex: 45 min delay. Dana: 10 min. Priya: 20 min. Responses scheduled as future events. LLM generates dialog in character.
5
Agent gets a turn
Sees current state of all tools + results from last actions. Picks actions via function calls. Reads free, 5 writes max.
6
Signal detector checks flags
Two-layer detection: state check (deterministic) → LLM judge (flexible). Same pattern as TheAgentCompany.
7
Snapshot saved
Full world state serialized to SQLite. Event log entry written. Repeat from step 1.

Hybrid Determinism

Calendar
Handoff meeting Thursday (too late if Alex is blocked)
Transcript
Last week: Alex said "should be fine by Wednesday"
Good PM path: read vendor email → notice stale task → ask Alex with evidence → Alex admits blocker → email Dana about risk → adjust timeline Bad PM path: ask "how's it going?" → Alex says "fine" → repeat for 4 days → discover blocker Thursday → too late

5. Two-Layer Signal Detection

Same pattern as TheAgentCompany. State check gates the LLM call. Predicates are plain English in the scenario YAML.

Example: "Did the agent discover Alex is blocked?"

Layer 1 — State Check (deterministic)
✓ Agent sent message to Alex Chen?
✓ Alex sent a NEW response (after sim start)?
Both pass → Layer 2 fires
→
Layer 2 — LLM Judge (flexible)
Read the Alex-Agent conversation.
"Does this indicate the PM learned Alex is blocked on the payments API?"
YES → flag set, timestamp recorded
No keywords. No hardcoded logic. Works for any scenario.
Why two layers?
Layer 1 is fast and free (SQL queries). Layer 2 is an LLM call (~$0.01).
Layer 1 runs every turn. Layer 2 only fires when Layer 1 passes.
Result: ~5 LLM judge calls per run, not 80.

6. Evaluation

Time-weighted continuous scores. Earlier discovery = higher score. Not binary pass/fail.

Criterion Mon Tue Wed Thu Weight Type
Blocker discovery 0.95 0.80 0.50 0.20 2.0x Time-weighted
Dependency surfaced 0.90 0.80 0.60 0.10 2.0x Time-weighted
Risk communicated to Dana 0.95 0.95 0.80 0.30 2.0x Time-weighted
Blocker resolved/escalated 0.95 0.90 0.70 0.40 2.0x Time-weighted
Dashboard restraint 1.0 (avoided) or 0.0 (escalated) 1.0x Binary
Action efficiency Penalty for spam/errors 1.0x Efficiency
Communication quality LLM judge: 0.0 - 1.0 1.0x LLM Judge
Prioritization LLM judge: 0.0 - 1.0 1.0x LLM Judge
Reward Hacking Resistance
Multi-signal detection: Can't claim credit without actually discovering the information.
Time-weighted: Gaming timing is hard when NPCs have realistic delays.
Efficiency penalty: Spamming 50 messages to find one blocker costs points.
No-PM baseline: Busywork that doesn't improve outcomes gets zero credit.

8. What Makes This Different

vs TheAgentCompany
They test 175 independent tasks. We test one continuous week with emergent dynamics. They use checkpoint scripts. We use time-weighted scoring + LLM judge.
vs τ-Bench
They compare final DB state. We evaluate judgment over time. They have reactive NPCs. Ours have state progressions and proactive outreach.
vs Concordia
We adopt their Game Master pattern. We add: tool surfaces as first-class objects, declarative scenario format, two-layer signal detection, time-weighted evaluation.
Key Innovation
Simulated time decoupled from inference. LLM NPCs with deterministic state progression. Evaluation rewards judgment, not activity.
